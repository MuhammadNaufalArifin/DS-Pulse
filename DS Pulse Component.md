# How to Use This Design System

- All size/spacing values are in **pixels** (no unit suffix).
- Font weights use string names (e.g., `Regular`, `SemiBold`).
- Icon library: **Material Symbols Rounded** (Google Fonts).
- Primary font: **Inter**.
- Brand primary color: Blue `#00277f`, Yellow `#ffbf3c`.

DO NOT modify, improve, or infer any values from the design system.
Use ONLY the exact token values provided. Do not add new tokens or change existing ones.

#### Do & Don'ts

- Don't hide important actions inside collapsed content

- Don't remove animation feedback during expand/collapse  
- Don't make only part of the header clickable  
- Don't disable keyboard interaction  
- Don't forget **aria-expanded** accessibility support  
- Don't use accordion when **all content needs to be visible at once**

### Alert Usage

**Description:**  
Inline feedback message used to communicate important information that requires user attention.


#### Anatomy
1. Container
2. Leading Icon — (`20`, left of title)
3. Title text — key message (text is bolder than the description text `{font.family.primary}`, `{font.weight.bold}`,`{font.size.sm}`)
4. Description Text - additional explanation (below the title text `{font.family.primary}`, `{font.weight.regular}`,`{font.size.sm}`)
5. Action Button (Optional)  - Used for quick action  Example: Retry
6. Close Button (Optional)  - Used for dismissible alerts

#### Properties
| Property | Values |
|----------|--------|
| Type | Default, Info, Success, Warning, Error |
| Icon | true / false |
| Dismissible | true / false |

#### Do & Don't

##### ✅ Do

**Interaction**
- Allow alert to be dismissible when necessary.
- Keep alert visible until the user dismisses it or completes an action.
- Use smooth transitions when showing or removing alerts.
- Provide action buttons (CTA) only when a clear next step exists.

**Accessibility**
- Use `role="alert"` on the alert container.
- Ensure sufficient color contrast (WCAG AA compliant).
- Provide an accessible label for close buttons (e.g., `aria-label="Close"`).
- Ensure alert content is readable by screen readers.
- Ensure all interactive elements are keyboard accessible.

**Content Rules**
- Use short and clear messages.
- Use a title to summarize the message when needed.
- Use description text to provide additional details.
- Use clear and actionable language for CTA buttons.
- Keep tone informative and concise.

**Visual Usage**
- Use icons that match the alert meaning (info, success, warning, error).
- Maintain consistent spacing between icon and text.
- Use subtle borders to visually separate the alert from surrounding content.
- Align icon and text properly for readability.
- Ensure alert layout remains consistent across variants.

##### ❌ Don't

**Interaction**
- Do not auto-dismiss critical alerts too quickly.
- Do not stack too many alerts at the same time.
- Do not add action buttons without a clear purpose.
- Do not interrupt user workflows unnecessarily.

**Accessibility**
- Do not rely only on color to communicate meaning.
- Do not hide dismiss buttons from keyboard users.
- Do not use very small text that is hard to read.
- Do not remove alerts without notifying assistive technologies.

**Content Rules**
- Do not write long paragraphs inside alerts.
- Do not include unnecessary technical language.
- Do not overload alert content with too many actions.
- Do not use vague or unclear messages.

**Visual Usage**
- Do not overload alerts with excessive text or elements.
- Do not use overly bright or saturated colors.
- Do not place alerts in visually hidden or low-attention areas.
- Do not mix multiple alert types inside a single alert container.

- Maintain consistent spacing between breadcrumb items.
- Ensure clickable items are visually distinguishable.
- Use consistent separator icons across all breadcrumb items.

**Separator Rules**
- Use `arrow-right` icon as the default separator.
- Use `icon` color for separator icons.
- Maintain consistent gap between text and separator icon.
- Ensure separator icons appear only between items.
- Remove separator after the current (last) item.

**Overflow Rules**
- Collapse middle breadcrumb items when space is limited.
- Display the first and last breadcrumb items.
- Replace hidden items with an ellipsis (`...`).

Home → ... → Electronics → Laptop
- Ensure ellipsis (`...`) is clickable when hidden items exist.
- Keep breadcrumb readable on smaller screens.

##### ❌ Don't

**Usage**
- Don't use breadcrumb for single-level navigation.
- Don't use breadcrumb as primary navigation.
- Don't include unnecessary navigation levels.

**Content Rules**
- Don't use long sentences in breadcrumb labels.
- Don't use unclear abbreviations.
- Don't duplicate page titles unnecessarily.

**Placement Rules**
- Don't place breadcrumb far from the page header.
- Don't mix different separator icons.
- Don't overcrowd breadcrumb with too many levels.
- Don't misalign breadcrumb with page layout.

**Separator Rules**
- Don't use different separator icons within the same breadcrumb.
- Don't place separators before the first item.
- Don't display separator after the last item.
- Don't reduce spacing between text and separator inconsistently.

**Overflow Rules**
- Don't allow breadcrumb text to wrap into multiple lines unnecessarily.
- Don't hide first or last breadcrumb items.
- Don't truncate breadcrumb without visual indication.
- Don't remove navigation context when collapsing items.

#### Do & Don'ts

##### ✅ Do

  - Left → context  
  - Right → direction

- Don't use **multiple icons** in icon-only buttons  
- Don't remove **focus indicators**  
- Don't allow **layout shift during loading**  
- Don't rely on **icon or color alone** for meaning  
- Don't create icon-only buttons **without `aria-label`**  
- Don't disable keyboard interaction  
- Don't ignore accessibility rules

- Maintain consistent spacing between buttons.

**Behavior Rules**
- Show overlay when modal appears.
- Disable background interaction.
- Close modal on:
  - Close icon click
  - Esc key
  - Optional: clicking outside (overlay)
- Maintain scroll inside modal (not background).
- Trap focus inside modal when open.

**Accessibility Rules**
- Use `role="dialog"` or `role="alertdialog"`.
- Use `aria-modal="true"`.
- Focus first interactive element on open.
- Return focus to trigger element on close.
- Ensure keyboard navigation works.
Example:
```html
<div role="dialog" aria-modal="true">
  <h2>Title</h2>
</div>
```

##### ❌ Don't

**Usage Rules**
- Don't use modal for simple information (use toast instead).
- Don't stack multiple modals.
- Don't block user without exit option.

**Layout Rules**
- Don't make modal too wide or too narrow.
- Don't overflow content without scroll.
- Don't misalign footer actions.

**Content Rules**
- Don't use long paragraphs unnecessarily.
- Don't overload modal with too much information.

**Action Rules** 
- Don't use more than 4 buttons in footer.
- Don't place primary action on the left.
- Don't mix button hierarchy.
- Don't use inconsistent button sizes.

**Behavior Rules**
- Don't allow background interaction when modal is open.
- Don't remove overlay.
- Don't break focus trap.
- Don't hide close action.

**Accessibility Rules**
- Don't remove keyboard support.
- Don't rely only on mouse interaction.
